# KNOW_YOUR_ENTITY — Core concepts

This document contains the core conventions and concepts used across the project. It is intended to provide high-level guidance and point to dedicated guides for platform-specific details.

See also:

- Frontend: `KNOW_YOUR_FRONTEND.md` (HTML / CSS / JS)
- Backend / API: `KNOW_YOUR_BACKEND.md` (REST API design)

## Структура сущностей

Вложенность не всегда плоха — в практике допустимы 1–2 уровня вложенности для удобства представления данных. При этом важно различать:

- представления/DTO, используемые для UI или ответов API (где можно допускать лёгкую вложенность для удобства клиента),
- и базовую доменную модель (сущности), которая должна оставаться как можно более плоской.

Правило простыми словами:

- 1–2 уровня вложенности разрешены в агрегированных представлениях и DTO.
- Основные доменные сущности (например, `User`, `Article`, `Product`) должны быть строго плоскими: поля сущности — примитивы или идентификаторы ссылочных сущностей (`userId`, `authorId`), без вложенных объектов, содержащих полные модели других сущностей.
- Избегайте глубокой вложенности (3+ уровней) в основной модели — она усложняет обновления, валидацию, поиск и миграции.

Когда стоит применять вложенность:

- Для удобства клиента в ответах API (например, включить краткую информацию об авторе в комментарии).
- Для временных представлений (проекции, view models), которые не являются источником правды в БД.

Когда использовать ссылки/денормализацию:

- Если вам нужно сохранить однозначный источник правды и упростить обновления — используйте идентификаторы (`authorId`) и храните связанные объекты отдельно.
- Денормализация допустима ради производительности, но должна быть осознанной и документированной.

Примеры:

Неправильно (основная сущность `User` содержит вложенный `profile` — запрещено для доменной модели):

```json
{
  "id": 2,
  "profile": {
    "age": 30,
    "bio": "..."
  }
}
```

Правильно (плоская доменная модель, профиль как отдельный ресурс или поля вынесены):

```json
{
  "id": 2,
  "age": 30,
  "bio": "..."
}
```

Допустимо в ответе API (агрегированное представление, 1 уровень вложенности):

```json
{
  "id": 1,
  "title": "Post",
  "comments": [
    {
      "id": 10,
      "text": "Nice",
      "author": { "id": 2, "name": "Bob" } // 1 уровень вложенности — DTO, не доменная модель
    }
  ]
}
```

Пример выше — корректный уровень вложенности (1–2 уровня). Если же вложенность возрастёт (3+ уровней), это существенно увеличит когнитивную нагрузку и усложнит поддержку, поэтому такие случаи следует избегать в доменной модели.

Итого: допускайте небольшую вложенность в представлениях и DTO (1–2 уровня), но держите основные сущности (`User`, `Article` и т.д.) плоскими — это облегчает поддержку, миграции и частичные обновления.

## Выделение сущностей и повторное использование

`User` — это плоский набор полей, представляющий основную доменную сущность. Если у пользователя есть адрес, который с высокой вероятностью будет переиспользоваться (например, для доставки, выставления счётов, офиса и т.д.), то адрес стоит вынести в отдельную сущность `Address`/таблицу. Выделенные сущности обеспечивают строгую структуру данных и облегчают переиспользование.

Отсюда следует практическое правило: выделяйте сущности (таблицы) для концептов, которые повторяются или имеют собственную жизненную логику/валидацию.

Возвращаясь к `User`, передача объекта в ответе может выглядеть двумя способами в зависимости от контекста эндпойнта:

```http
// GET /profile/{id}
{
  "user": {
    "contact": {}, // email, phone
    "address": {}
  }
}
```

или

```http
// GET /user/{id}
{
  "contact": {},
  "address": {}
}
```

Разница в пути и форме ответа важна: путь эндпойнта подсказывает, какую сущность мы возвращаем. `/profile/{id}` обычно означает агрегированное представление (profile = набор данных о пользователе для UI), тогда логично вернуть объект `user` внутри обёртки. `/user/{id}` более явно запрашивает саму доменную сущность `User`, и тогда тело ответа лучше отражает границы этой сущности (поля сверху). Выбирайте URL и структуру ответа так, чтобы они явно передавали границы сущностей и ожидания клиента.

## Именование переменных, объектов, методов, классов и файлов

Хорошая система именования делает код понятным, последовательным и простым в поддержке. Ниже — рекомендации и практические примеры, которые помогут выработать единый стиль для команды.

## Общие принципы

- Имя должно отражать сущность и её назначение, а не реализацию (не включайте типы, формат или технические детали).
- Отдавайте предпочтение ясным и полным именам: читаемость важнее экономии нескольких символов.
- Следуйте одному стилю (camelCase для переменных/функций, PascalCase для классов/интерфейсов, kebab-case или camelCase для файлов в зависимости от проекта).
- Согласуйте правила с командой и зафиксируйте их в стиле репозитория.

## Именование переменных

### Примитивы

#### Не указывайте тип в имени

Не дублируйте тип в названии переменной — это добавляет шум и усложняет поддержку. Типы должны быть в аннотации или очевидны по контексту и IDE.

```ts
// BAD
const ageNumber: number = 30;

// GOOD
const age = 30;
```

#### Предпочитайте понятные полные имена

Избегайте неопределённых сокращений (`res`, `tmp`, `data`) — лучше использовать `response`, `result`, `users` и т.д. Если контекст ясен (короткая функция, очевидный смысл), локальные сокращения допустимы (`u` в мапе `users.map(u => u.name)`).

```ts
// BAD
const a = 30;

// GOOD
const age = 30;
```

Пример, когда разницы между `result` и `response` важна:

```ts
const getUserNames = async () => {
  const result: string[] = [];
  const response = await fetch('/users');
  const users = await response.json();
  if (users?.data) result.push(...users.data.map(user => user.name));
  return result;
}
```

### Сокращения

Используйте сокращения экономно и только если смысл понятен из контекста (например, `id`, `msg`, `err`). Если это названия общего назначения (например, `err`), то лучше сделать их частью руководства по стилю. Не вводите свои аббревиатуры без объяснения.

## Именование объектов и классов

Класс или интерфейс описывает сущность и её свойства. Наиболее распространённый пример — `User`.

```ts
class User {
  id: number;
  name: string;
}
```

### Не дублируйте имя сущности в её свойствах

Не называйте поля `userId`, `userName` внутри `User` — это излишне.

```ts
// BAD
class User {
  userId: number;
  userName: string;
}

// GOOD
class User {
  id: number;
  name: string;
}
```

Чаще всего такие ошибки совершаются при передачи данных объекта DTO. Например:

```json
// GET /user/{id}
{
  "userName": "Bob",
  "userEmail": "bob@bob.bo
}
```

Тем временем, как можно использовать:

```json
// GET /user/{id}
{
  "name": "Bob",
  "email": "bob@bob.bo
}
```

Если мы передаём несколько сущностей, то:

```json
// GET /user/{id}
{
  "user": { // User
    "name": "Bob",
    "email": "bob@bob.bo"
  },
  "address": { // Address
    "city": "City",
    "zipCode": "111111"
  }
}
```

Такие объекты всегда проще типизировать.

### Параметры в виде объектов

Если у вас набор параметров — оформляйте отдельный интерфейс/тип. Используйте единственное число для типа и множественное для коллекций.

```ts
interface Filter {
  priceFrom?: number;
  priceTo?: number;
}

const filter: Filter = { priceFrom: 0, priceTo: 99 };
const filters: Filter[] = [filter];
```

Если нужен именованный фильтр:

```ts
interface NamedFilter {
  name: string;
  valueFrom?: number;
  valueTo?: number;
}

const priceFilter: NamedFilter = { name: 'price', valueFrom: 0, valueTo: 99 };
```

## Именование функций и методов

Выбирайте глаголы, которые отражают семантику действия:

- `get` — вернуть существующее значение (обычно синхронно или из кэша).
- `set` — установить или изменить значение.
- `find` — найти элемент по критериям, возможно вернуть `null`/`undefined`.
- `fetch` / `load` — загрузить данные (обычно сетевой запрос, асинхронно).
- `create` / `update` / `delete` — CRUD-операции.

```ts
function getAppUser(): User | undefined {}
function setAppUser(user: User): void {}
function findUser(filters: Filter[]): User | null {}
async function fetchUsers(): Promise<User[]> {}
```

Старайтесь, чтобы имя метода было предсказуемым и единообразным в проекте.

## Именование классов

Для компонентов и сервисов используйте понятные суффиксы: `Controller`, `Service`, `Repository`. Если принято сокращать — согласуйте сокращения с командой.

```text
UserController -> UserCtrl (если сокращаете)
UserService  -> UserSvc
```

Лучше предпочитать полные имена (`UserController`) ради ясности.

## Именование файлов

Названия файлов должны отражать содержимое. Для сущностей обычно используют единственное число и имя класса.

```text
// user.ts
class User {}
```

Для view/страниц или компонентов можно использовать имя маршрута или функциональности: если маршрут `/users`, компонент может называться `users.component.ts`.

Выбирайте согласованный стиль (kebab-case, snake_case или camelCase) и применяйте его в проекте.

## Примеры и антипаттерны

1. Плохой пример — дублирование типа и сущности в имени:

```ts
// BAD
const userListArray: User[] = [];
```

1. Хороший пример — ясные имена и разделение ответственности:

```ts
const users: User[] = [];
function findUserByName(name: string) { return users.find(u => u.name === name); }
```

- [ ] Имя отражает сущность и её назначение
- [ ] Не указан тип в имени (например, `countNumber`)
- [ ] Нет избыточного дублирования (`user.userId` внутри `User`)
- [ ] Использованы понятные глаголы для действий (`get`, `find`, `fetch`, `create`)
- [ ] Имена файлов соответствуют содержимому и общему стилю
